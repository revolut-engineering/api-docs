    Merchant API

    Introduction to Merchant accounts

    Please note that integrating the Merchant API requires a Merchant Account which is available only to businesses that have been invited to participate in our private launch. In the meantime, our team is working hard to make it available to every eligible business. Stay tuned!

    Revolut has created Merchant Accounts to provide customers with an opportunity to accept online debit & credit card payments straight into their Revolut Business Merchant accounts.

    There are two ways to accept card payments:

    • Checkout with Revolut Widget: if you have your own e-commerce website, you can use either a payment pop-up or a card field to add into your checkout page. This will require integration with Revolut Backend API.

    • Checkout with official plugins: if you already have a website running on one of our supported e-commerce platforms, you can install a respective plugin which Revolut has created. At the moment we support WooCommerce, Prestashop and Magento. Stay tuned for more plugins becoming available soon.

    You can decide which one of those checkout methods suits your website and proceed to the respective guide to start the integration!

    Integration guide

    This guide will help you to set up a Revolut Checkout Widget and configure your back-end infrastructure to start accepting online debit & credit card payments on your website.

    The Checkout Widget can be integrated in two different ways:

    • as a pop-up window appearing over the currently open web-page and requesting the payer's card information;
    • as a card field which is integrated as a separate object on your checkout page.

    This page provides a general overview of how to integrate the Checkout Widget. For more detailed documentation, please refer to the following guides:

    First Steps

    Step 1. In your Revolut Business account, apply for a Merchant Account. Once your application gets approved, get an API token.

    Step 2. Create a new order on your server back-end with the amount and currency.

    Create a new order

    POST https://merchant.revolut.com/api/1.0/orders

Authorization: Bearer <your_API_token>

{
  "amount": 200,
  "currency": "GBP"
}

    Response

    {
  "id": "ID",
  "public_id": "TOKEN_XXX",
  "type": "PAYMENT",
  "state": "PENDING",
  "created_date": 0,
  "updated_date": 0,
  "order_amount": { "value": 10000, "currency": "GBP" }
}

    From the response, save the public_id: it is a temporary token generated for every new order a customer initiates on a merchant's website. Normally, a public_id is created at the moment when the amount is known for the order. A public_id is also the identifier which is passed from the back-end server to the client side in order to initiate the checkout widget. Once the order is paid, a public_id expires.

    You will need to create a new order for each purchase.

    Step 3. Add an embed script snippet to the html document head in the page template. This script will be requested asynchronously so you don't have to worry about content blocking the loading of the page.

    Embed script snippet

    <head>
  <title>Shop</title>
  <script>!function(e,o,t){e[t]=function(n,r){var c={sandbox:"https://sandbox-merchant.revolut.com/embed.js",prod:"https://merchant.revolut.com/embed.js",dev:"https://merchant.revolut.codes/embed.js"},d=o.createElement("script");d.id="revolut-checkout",d.src=c[r]||c.prod,d.async=!0,o.head.appendChild(d);var s={then:function(r,c){d.onload=function(){r(e[t](n))},d.onerror=function(){o.head.removeChild(d),c&&c(new Error(t+" is failed to load"))}}};return"function"==typeof Promise?Promise.resolve(s):s}}(window,document,"RevolutCheckout");</script>
</head>

    The script also availble as NPM package for your convenience.

    NPM package

    npm install @revolut/checkout

    Step 4. Collect card details.

    After inserting a code snippet into the document head, you have two options to collect a card number and make a payment:

    • Pop-up - the easiest and fastest method to get started;
    • Card field - fits right in your custom designed checkout flow

    Below you will find detailed instructions on how to set up either of those methods.

    Pop-up

    A pop-up window is the easiest method to accept payments with Revolut. See example below:

    <button id="pay-button">
  Pay
</button>

<script>
  RevolutCheckout("TOKEN_XXX").then(function(RC) {
    var payButton = document.getElementById("pay-button");

    // On click open payment pop-up
    payButton.addEventListener("click", function() {
      RC.payWithPopup({
        // (optional) name of the customer
        name: "First Last",
        // (optional) email of the customer
        email: "customer@example.com",
        // (optional) phone of the customer
        phone: "+447950630319",
        // (optional) billing address of the customer
        billingAddress: {
          countryCode: "UK",
          region: "Greater London",
          city: "London",
          streetLine1: "Revolut",
          streetLine2: "1 Canada Square",
          postcode: "EC2V 6DN"
        },
        // (optional) shipping address of the customer
        shippingAddress: {
          countryCode: "UK",
          region: "Greater London",
          city: "London",
          streetLine1: "Revolut",
          streetLine2: "1 Canada Square",
          postcode: "EC2V 6DN"
        },
        // Callback called when payment finished successfully
        onSuccess() {
          window.alert("Thank you!");
        },
        // Callback in case some error happened
        onError(message) {
          window.alert("Oh no :(");
        },
        // (optional) Callback in case user cancelled a transaction
        onCancel() {
          window.alert("Payment cancelled!")
        },
      });
    });
  });
</script>

    The shown snippet will open a payment pop-up window when a user clicks on the payment button. So, you need to create a button and attach a click event listener to it that will trigger RC.payWithPopup(options).

    Card Field

    Inserting a card field into your checkout page allows you to create a separate field securely accepting card details from the users and to submit payments with the rest of your payment form.

    <form id="form">
  <div id="card-input"></div>
  <button>Submit</button>
</form>

<script>
  RevolutCheckout('TOKEN_XXX').then(function(RC) {
    var form = document.getElementById('form')
    var cardInput = document.getElementById('card-input');

    var card = RC.createCardField({
      // empty `<div>` inside your form
      target: cardInput,
      // callback called when payment finished successfully
      onSuccess() {
        window.alert('Thank you!')
      },
      // callback in case error happened
      onError(message) {
        window.alert('Oh no :(')
      },
      // callback in validation status change
      onValidation(messages) {
        console.log(messages) // -> [ 'Your card has expired' ]
      }
      // (optional) Callback in case user cancelled a transaction
      onCancel() {
        window.alert("Payment cancelled!")
      },
      // (optional) see API reference
      styles: {},
      // (optional) see API reference
      classes: {},
    })

    form.addEventListener('submit', function(event) {
      event.preventDefault()

      /**
        Gather additional info from the form if needed
        and submit card number to finish the order
      */

      card.submit({
        // (optional) name of the customer
        name: 'First Last',
        // (optional) email of the customer
        email: 'customer@example.com',
        // (optional) phone of the customer
        phone: '+447950630319',
        // (optional) billing address of the customer
        billingAddress: {
          countryCode: 'UK',
          region: 'Greater London',
          city: 'London',
          streetLine1: 'Revolut',
          streetLine2: '1 Canada Square',
          postcode: 'EC2V 6DN',
        },
        // (optional) shipping address of the customer
        shippingAddress: {
          countryCode: 'UK',
          region: 'Greater London',
          city: 'London',
          streetLine1: 'Revolut',
          streetLine2: '1 Canada Square',
          postcode: 'EC2V 6DN',
        },
      })
    })
  });
</script>

    The snippet shows how you can create a form with a div element in it which will render a card field and then submit a payment.

    Saving card details and creating customers

    You can save the card details of your customers to be able to charge them later. There are two different processes that will allow you to do this:

    Customer is created first and then attached to order

    1. Create a customer
    2. Create an order with the amount, currency and customer_id parameter in the body of the request
    3. Make payment via the widget with options.savePaymentMethodFor = customer or merchant
    4. Payment method will be created and attached to the customer

    Customer is created as part of order completion

    1. Create an order with amount and currency
    2. Make payment via the widget with options.savePaymentMethodFor = customer or merchant
    3. Either the order or widget should have customer_email (otherwise there will be an error returned)
    4. If payment is successful, Customer is created with email and other details
    5. Payment method is associated to this customer
    6. customer_id will be attached to the order automatically

    Charging a customer's saved payment method

    Once you've created a customer and successfully saved a payment method, you can charge this payment method without the customer being present in the checkout page (e.g., for subscription services). You can also use this to speed up the checkout process when the customer is present on the checkout page (e.g., for 1-click buy flows).

    To do so, just follow these two steps:

    Create Order

    POST /api/1.0/orders

{
  "amount" : 100,
  "currency" : "GBP",
  "customer_id" : "6c7c97a8-cfc1-4cf3-8b38-26a74fdf1fae"
}

    Confirm Order

    
POST /api/1.0/orders/6a1353a8-3054-40ee-ab39-97a11e4c5f2a/confirm
    1. Create an order with the amount, currency and customer_id parameter in the body of the request
    2. Confirm the order to take the order amount from this customer

    Note: If payment_method_id is not specified in the body of the order confirmation request, payment will be taken from the payment method that was last saved for this customer.

    RevolutCheckout

    type RevolutCheckout = (
  token: string,
  mode?: 'prod' | 'sandbox' = 'prod'
) => Promise<Instance>;

    Load the embedded script and create an Instance for the order token.

    Parameters

    Script load example

    RevolutCheckout("XXX", "prod").then(function (instance) {
  // Work with instance:
  // instance.payWithPopup()
  // instance.createCardField()
  // ...
});
    • token - public_id returned from the API request
    • mode - (optional) API environment either 'prod' or 'sandbox', defaults to 'prod'

    Returns Promise with order Instance.

    Instance

    type Instance = {
  payWithPopup: InstancePayWithPopup;
  createCardField: InstanceCreateCardField;
  destroy: () => void;
};

    Instance is passed as an argument to the consuming function of the RevolutCheckout promise e.g., registered using .then. It can be used to open a payment pop-up window or to create a card field.

    Instance usage example

  instance.payWithPopup({
    onSuccess() {

    }
  });
});

    Instance.payWithPopup

    Open a payment pop-up which is available on the Instance.

    type InstancePayWithPopup = (
  options: Options
) => {
  destory: () => void;
};

type Options = {
  onSuccess?: () => void;
  onError?: (message: string) => void;
  onCancel?: () => void;
  name?: string;
  email?: string;
  phone?: string;
  billingAddress?: {
    countryCode: string;
    region: string;
    city: string;
    postcode: string;
    streetLine1: string;
    streetLine2: string;
  };

    countryCode: string;
    region: string;
    city: string;
    postcode: string;
    streetLine1: string;
    streetLine2: string;
  };

    Options object:

    Field Description Format
    options.onSuccess Callback when the payment is successfully completed Function
    options.onError Callback if the transaction fails to complete. The reason for the failure should be available in the message parameter Function
    options.onCancel Callback if a user did not complete the transaction and cancelled the authorisation or closed the payment pop-up window Function
    options.name Customer name in form of FirstName LastName String
    options.email Customer's email String
    options.phone Customer's phone number if available String
    options.savePaymentMethodFor Indicates that the payment method will be saved for future use against the customer’s profile.

    Possible values:
    - customer: method will be saved for being used in the future when the customer is on your checkout page
    - merchant: method will be saved for being used in the future without needing the customer to be on your checkout page (e.g., to take payments for recurring transactions). Method will be saved as the new default payment method for merchant initiated transactions, regardless of the number of saved payment methods stored against the customer    		 Enum
    options.billingAddress Contains customer's billing address which is required if not set on order via API Object
    options.billingAddress.countryCode Country code (e.g. GB) String
    options.billingAddress.region State, county or region String
    options.billingAddress.city Name of the city (e.g. London) String
    options.billingAddress.postcode Postal code (e.g. EC2V 6DN) String
    options.billingAddress.streetLine1 Address line 1 (e.g. 1 Canada Square) String
    options.billingAddress.streetLine2 Address line 2 (optional) String
    options.shippingAddress The same object as billingAddress object, however, it is only displayed in the order details on the merchant dashboard Object

    Returns

    Methods object:

    Field Description Format
    methods.destroy Manually destroy payment pop-up if needed Function

    API Reference

    Introduction

    The Orders API allows Revolut Business customers to accept online (Card Not Present) payments made with debit/credit cards, and, subsequently, to manage these transactions by means of capturing, canceling or refunding orders.

    All merchant requests should contain an authorisation header which includes the name of the merchant account Bearer and the MERCHANT_API_KEY.

    You can generate the MERCHANT_API_KEY on a merchant portal. In order to do so, please proceed to the Merchant tab in your Revolut Business app, click on the API menu option, and you will find the MERCHANT_API_KEY under the section called Production API key.

    All requests and responses should have a JSON format.

    Authorization

    curl
--header 'authorization: Bearer AUTHORIZATION_TOKEN'
--header 'content-type: application/json'

    You are advised to create an Order for every customer session or order being created in your own system. An Order will then go through multiple status as it interfaces with the Revolut Checkout Widget to collect payments from your customers. An Order can have at most one successful payment.

    Customers

    Customer objects allow you to track multiple charges associated with the same customer.

    Customers will be created automatically upon creating an order if there is enough information and no Customer ID is passed. Aditionally, the API allows you to create, delete, and update your customers. You can retrieve individual customers as well as a list of all your customers.

    Payment methods reference a method of payment that can be used to make transactions. It allows the merchant to store the details for payment securely and to group transactions that come from the same payment method. These payment methods are unique per customer and per merchant. So if, for example, the same card is used for a transaction for two different customers, two payment methods objects will be created.

    PaymentMethod’s can’t be created explicitly, they are generated as part of a payment. To modify a customerPaymentMethod, change its parameters using a customer id and sending the appropriate parameters

    Customer object

    Attribute Description Type
    id Unique identifier for the object UUID
    full_name Customer’s full name. If customer is not created with this parameter, full_name will be taken from the cardholder name the first time a payment is made string
    business_name Customer’s business name string
    email Customer’s email address. This email needs to be unique for every customer for this merchant string
    phone Customer’s phone number (e.g. “+44 7666 888 999”) string
    created_at the instant when the customer was created ISO date/time
    updated_at the instant when the customer was last updated ISO date/time

    Intrinsically linked to the customer object. Payment Methods are unique per customer and per merchant. Meaning that if the same card is used across two of your customers, two paymentMethod will exist with two separate IDs.

    Request in cURL

    curl -X "GET" "https://merchant.revolut.com/api/1.0/customers/:CUSTOMER_ID/payment-methods" \
     -H 'Authorization: Bearer <your_API_token>' \
     -H 'Content-Type: application/json; charset=utf-8' \
}

    Response

    List of payment method objects

    You can use this request to retrieve all payment_methods for a given customer. This could be useful for example to show the customer what information is stored for them, or to try a different payment method in case the first payment method fails when charging a recurring transaction.

    Body Parameters

    Parameter Description Required?
    only_merchant Can be used to retrieve only the payment_methods that have been saved by a customer to be used for merchant initiated transactions optional (defaults to FALSE)

    Retrieve specific customer's payment method

    Request

    GET /api/1.0/customers/:CUSTOMER_ID/payment-methods/:paymentMethodId

    Request in cURL

    curl -X "GET" "https://merchant.revolut.com/api/1.0/customers/:CUSTOMER_ID/payment-methods/:paymentMethodId" \
     -H 'Authorization: Bearer <your_API_token>' \
     -H 'Content-Type: application/json; charset=utf-8' \
}

    Response

    Payment method object

    This request can be used to retrieve the information of a specific payment method that has been saved.

    Body Parameters

    This request does not accept any body parameters

    Update what a specific payment method is saved for

    Request

    PATCH /api/1.0/customers/:CUSTOMER_ID/payment-methods/:paymentMethodId

{
  saved_for: 'customer'
}

    Request in cURL

    curl -X "PATCH" "https://merchant.revolut.com/api/1.0/customers/:CUSTOMER_ID/payment-methods/:paymentMethodId" \
     -H 'Authorization: Bearer <your_API_token>' \
     -H 'Content-Type: application/json; charset=utf-8' \
     -d $'{
        "saved_for": "customer",
        }'

    Response

    Updated payment method object

    Use this request to indicate that a payment method can't be used for merchant initiated payments anymore and can only be used for customer initiated payments

    Body Parameters

    Parameter Description Type Required?
    saved_for Indicate that a customer doesn’t want a payment method to be used for merchant initiated transactions anymore but yes for future use (merchant → customer) enum required

    Delete specific customer's payment method

    Request

    DELETE /api/1.0/customers/:CUSTOMER_ID/payment-methods/:paymentMethodId

    Request in cURL

    curl -X "DELETE" "https://merchant.revolut.com/api/1.0/customers/:CUSTOMER_ID/payment-methods/:paymentMethodId" \
     -H 'Authorization: Bearer <your_API_token>' \
     -H 'Content-Type: application/json; charset=utf-8' \
}

    Response

    Success HTTP response with no body

    This request can be used to delete a saved payment method. It will be completely deleted from the customer payment methods. If you would like to use it again, direct your customer to the checkout and save the card details again.

    Body Parameters

    This request does not accept any body parameters

    Webhooks

    There are multiple events happening to Merchant Accounts. The majority of these occur synchronously - this means they arrive instantly and an immediate result is available. For example, an Order creation request is a synchronous request.

    However, there are some events that run asynchronously and, therefore, require webhooks. Merchant API supports webhooks for order completion events.

    Set webhook url

    This request can be used by a merchant to subscribe for webhooks from Revolut Merchant API. Webhooks will push information to the specified URL

    Request

    POST /api/1.0/webhooks

{
  "url": "http://your-company.com/webhook",
  "events": [
    "ORDER_COMPLETED",
    "ORDER_AUTHORISED"
  ]
}

    Response

    [
  {
    "id": "d3e22dd1-ad0c-49aa-b99b-1f047abf8b82",
    "url": "http://your-company.com/webhook",
    "events": [
      "ORDER_COMPLETED",
      "ORDER_AUTHORISED"
    ]
  }
]

    Request

    Field Description Format
    url valid URL if set, else null URL
    events list of events that the webhook is configured for

    Possible values are ORDER_COMPLETED and/or ORDER_AUTHORISED    		 list

    Response

    Field Description Format
    id id of the webhook string
    url valid URL if set, else null URL
    events list of events that the webhook is configured for

    Possible values are ORDER_COMPLETED and/or ORDER_AUTHORISED    		 list

    Retrieve webhooks

    This request is used to get the list of webhooks you are currently subscribed to.

    Request

    GET /api/1.0/webhooks

    Response

    [
  {
    "id": "d3e22dd1-ad0c-49aa-b99b-1f047abf8b82",
    "url": "http://your-company.com/webhook",
    "events": [
      "ORDER_COMPLETED",
      "ORDER_AUTHORISED"
    ]
  }
]
    Field Description Format
    id id of the webhook string
    url valid URL if set, else null URL
    events list of events that the webhook is configured for

    Possible values are ORDER_COMPLETED and/or ORDER_AUTHORISED    		 list

    Webhook request

    Revolut will POST webhook messages to the URLs you have set webhooks for. Here is an example of an order completion event.

    Request

    POST http://your-company.com/webhook

    Message

    {
  "event": "ORDER_COMPLETED",
  "order_id": "4ee2a044-4965-4617-96e7-9debea935dfc"
}
    Field Description Format
    event Name of the event that is being sent.

    Possible values are ORDER_COMPLETED or ORDER_AUTHORISED    		 string
    order_id Order ID of the order that has changed status UUID

    Installation

    Step 1. Log in to the admin section of your Prestashop 1.7 webstore.

    Step 5. Drop the plugin zip file that you just downloaded or select it using the file browser

    Step 6. Wait for the installation to finish. It should not take long.

    Step 7. Once finished, you're ready to go. Click the Configure button to set up the plugin

    Configuration

    To configure the Revolut Gateway for Prestashop 1.7 plugin log in to the admin section of your Prestashop webstore. On the left panel, locate the IMPROVE section, click on Modules and then Module Manager. Scroll down to the Other section and you should find the plugin.

    Settings explained

    • Enable plugin: Switch to Yes to enable the Revolut Payment Gateway on your Prestashop site
    • Title: Enter the gateway title which will be displayed at the customer’s checkout page
    • Description: Enter the description which will be displayed at the customer’s checkout page
    • Mode: Switch the payment gateway mode to Sandbox or Live. API Key (sandbox mode) or API Key (live mode): API Key from your Merchant settings on Revolut.
    • Enable "Authorize Only" mode...: Tick on this field if you want to enable the Authorize Only mode. If this field is left unchecked, the module will stay in Authorize and Capture mode
      • When ticked : check the card for validity but not charge the account yet until the order is approved and invoiced.
      • When unticked : charge the debit/credit card at the time of the order is placed

    Advanced mode

    • Customize your order status: You can customize your orders' status for each possible action in the Revolut payment module. If set to Yes, the following two options will appear:
      • Order Status for triggering the refund on Revolut: Change the status that triggers when an order is refunded Default status : Refunded
      • Order Status for triggering the capture on Revolut: Change the status that triggers when an order is captured Default status : Payment accepted

    Operations

    Capture an order from Prestashop's admin panel

    The integration of this plugin allows you to handle the capture of the orders from Prestashop's admin panel. To do so, just follow these simple steps:

    1. Ensure you have the plugin set in "Authorize Only" mode
    2. When a customer places an order, the status of the order in Prestashop will be "Awaiting check payment" by default
    3. To capture the funds for a specific order, change the status to "Payment accepted"

    You can change which status captures the funds by modifying the settings as explained above in the Advanced mode section of the settings.

    Refund an order from Prestashop's admin panel

    After the funds of an order have been capture (either with the process detailed above or because the plugin was not in "Authorize Only" mode), you can refund the amount of an order using Prestashop's admin panel. To do so, change the status of the order to "Refunded".

    You can change which status refunds the order by modifying the settings as explained above in the Advanced mode section of the settings.

    • Add your API Key found in your Merchant account settings on Revolut Business portal

    • Click the Save button to confirm on the bottom left of this section to apply the changes

    Magento 2

    Revolut Gateway for Magento 2 plugin lets you accept online card payments via your Magento 2 e-store - in an easy and hassle-free fashion. The following guide will help you install and configure it.

    Installation

    Follow these steps to install the plugin manually:

    Terminal (Step 3)

       $ unzip revolutgatewayformagento2.zip

    Terminal (Step 4)

      $ php bin/magento setup:upgrade
  $ php bin/magento setup:di:compile
  $ php bin/magento setup:static-content:deploy

    Terminal (Step 5)

      $ php bin/magento cache:flush

    Step 1. Download the Revolut Gateway for Magento 2 .zip file.

    Step 2. Locate the /app/code directory inside the root directory (you might need to create it).

    Step 3. Extract the zipped file in the /app/code directory.

    Step 4. Open a terminal in the magento2 root folder and run the commands in the provided code block.

    These should install the plugin for you and take care of everything.

    Step 5. Once the steps above are finished, flush the Magento 2 cache using the admin panel or the terminal command on the right.

    Step 6. Finally, go back to your store's admin panel and navigate to Stores > Configuration > Sales > Payment Methods > Revolut. If all steps went well, you should be good to go!

    Troubleshooting
    • If you have troubles running the commands in Step 4:
      • Make sure that the PHP version your terminal is using is the same version that your Magento 2 store is using.
      • Make sure that PHP can use enough memory by setting the memory_limit variable in the /conf/php.ini file to more than 756MB
    • The plugin is only compatible with Magento versions 2.1.x, 2.2.x or 2.3.x

    Configuration

    In this section you can find how to configure the plugin.

    Required settings

    • Select Mode: Switch the payment gateway mode to Sandbox or Live.
    • API Key (sandbox mode) or API Key (live mode): API Key from your Merchant settings on Revolut.

    Additional Settings

    • Enable : Choose Yes to enable Revolut Gateway
    • Title : Enter the title of the payment method displayed on the checkout page
    • Payment Action : Admin can select between Authorize Only and Authorize And Capture
      • Authorize Only : check the card for validity but not charge the account yet until the order is approved and invoiced.
      • Authorize And Capture : charge the debit/credit card at the time of the order is placed.
    • Always hide postcode: Select No if you want to allow customers to fill in their postcode at card details field and vice versa. This field is set as No by default. Card Form when disable Show Postcode
    • Payment from Applicable Countries: Admin can select All Allowed Countries or Specific Countries
      • All Allowed Countries: Enable all allowed countries to make payment.
      • Specific Countries: Restrict some countries by unselecting them from the Payment from Specific Countries.
    • New Order Status : Set order status for new orders as Processing or Suspected Fraud
      • Processing: The payment is made successfully and an order invoice is issued.
      • Suspected Fraud: No invoice is issued.
    • Sort Order : The order of the payment method that will be displayed on the checkout page.
    • Minimum Order Total : If the order value is less than this figure, the payment method will not be displayed on the checkout page.
    • Maximum Order in Total: If the order value is bigger than this figure, the payment method will not be displayed on the checkout page.

    Operations

    Capture an order from Magento's admin panel

    The integration of this plugin allows you to handle the capture of the orders from Magento's admin panel. To do so, just follow these simple steps:

    1. Ensure you have the plugin set the Payment Action setting to "Authorize Only".
    2. When a customer places an order, the status of the order in Magento will be "Processing" by default
    3. Select the order that has been placed and click on Invoice on the top right of the order details view
    4. Scroll to the bottom and, under the Invoice totals section, make sure you have selected the Capture online option
    5. Click the Submit Invoice button to capture the funds for this order

    Refund an order from Magento's admin panel

    You can refund the amount of an order totally or partially using Magento's admin panel following these steps:

    1. Once the amount of an order has been captured and an invoice has been generated, select the invoice of the order you would like to refund
    2. In the top right corner, select Credit Memo
    3. Scroll down to the Items to Refund section and select what items you would like to refund
    4. Scroll further down and click the Refund button when you're ready to process the refund

    Errors

    The API uses the following error codes:

    CODE MEANING
    400 Bad Request -- Your request is invalid.
    401 Unauthorized -- Your access token is wrong.
    403 Forbidden -- Access to the requested resource or action is forbidden.
    404 Not Found -- The requested resource could not be found.
    405 Method Not Allowed -- You tried to access an endpoint with an invalid method.
    406 Not Acceptable -- You requested a format that isn't JSON.
    429 Too Many Requests -- You're sending too many requests.
    500 Internal Server Error -- We had a problem with our server. Try again later.
    503 Service Unavailable -- We're temporarily offline for maintenance. Please try again later.
